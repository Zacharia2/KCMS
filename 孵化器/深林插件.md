- 一个由自包含列表构成的生长式内容管理系统，是一张无限文档。一个无限大的，由树构成的深林。文档中每个块都是一个拥有属性的条目，同级块们构成一个列表，块可以包含子列表。块有两种类型笔记块或者文本块，笔记块指向一个笔记文件，文本块是为了适应无名称笔记片段。
- 在设计上，他是一个在去掉集合唯一性的树，并在该树上建图的抽象模型。这个树结构本身就是集合在另一个视角去掉唯一性的等价变换。而列表是构成这个树结构的基本要件。
- 这个树是借鉴卢曼编码，他非传统集合树（单金字塔结构），而是一个相关树，可以想象为多个金字塔构成的树状结构。每个树的末端结点可以是下一个子树的入口，只需要相关即可。在这个树中，元素或者卡片说须与前一条卡片有关联（上下，前后，有关联即可。）扩展了对目录或文件的认识
- 是一种，类似注意力、聚焦、集中的内容管理系统，帮助 pkmer 快速聚焦一些信息，以便于结构化、和这些信息的长久保持。相关的内容可以聚集起来，并且可以不断的生长，就像分形一样。这样到时候可以快速找到，重用。并且通过特定模型字段关联笔记和信息。
- 不仅仅是为了使笔记有序生长，而是更进一步的去发现让更多未知的事物浮现在眼前。
- 关键词是：主题阅读、主题、建构，知识发现，重构，渐进写作，细胞的聚集、砂砾聚集、聚沙成塔，生长、碳骨架。

- 系统架构是由 ink（墨迹语义空间）和 page（元素页面) 构成
	- Ink 包含：Acc（公开性）、Priority（优先级）、Block（中图编号）、Type（类型）、Edge（边）。
	- Page 包含：Link（链接）、Comment（评论）、Reference（外部参考）
	- 我为这些关键词做了定义，这些关键词将构成整个系统的运行子组件。
- 浮现-归纳环
	- 浮现到归纳
		- 从 daily note 开始浮现到归纳
		- 从 page element 开始浮现到归纳
	- 归纳到浮现
		- 从 page element 开始归纳到浮现
- 在时间发展上 daily note、pages、moctree（虚树），任意一个都可以作为开始的创作的入口。
	- 这系统的一切现在发现都可以简化等价为表格。之前查询是直接查询 ink 字符串
	- 从外部参考，到创作、到存储。
	- 无序列表集合有可能会通过第一性原理+相对根结点变成逻辑有序 tree
	- Element 自组织为 moctree，如果元素和现有 moctree 相关就直接放入。
- 我认为真正的写作，是开始的时候没有结构，到一定的积累，结构开始展现。然后开始了结构的迭代和剥复循环。所以我要把笔记在这个角度分成两个部分存储。我需要一个无拘无束难度几乎可以忽略不计就像已经手拿笔面前摆好白纸一样，立刻开始书写而没有取名和归类的阻碍。在自由无压写作面前有两座大山，命名和归类。无论命名还是归类都是事后的，动态的。
	- 系统流程设计
		- 创作阶段从 0 到 1（收件箱暂存区 Zephyr）
			- 从 daily note 到 element 的时间线分布聚合组织
			- 页面集合（pages）
			- 外部参考（inbox）
				- 中图存储或主题存储
				- 并不是所有的内容都值得放进笔记里面，笔记里面的内容应当是可迭代可被检验的，不过另一类材料质量也不错可以放进笔记里面作为备份或者素材或者类似论文之类的资料。从这个角度看笔记就可以分成两部分，一部分是我写的，另一部分是我能够用得着的资料，也就是在旧的东西上长出新的花来。
				- 笔记所需要的参考文档
				- 非笔记所需要参考->潜在队列参考
				- 闭合开环（有取舍）
		- 存储阶段从 1 到∞（常青笔记）
			- 元素= name（页面名：笔记内容整体的概括抽象）+（page+link+ref+ink+comment）（内容体）
			- 笔记成长的特点是从日常笔记中生成独立元素，独立元素形成元素群，元素们相互联系起来。其实就是点集树的过程。
			- 聚点，类别，常青树（存储区必须有索引，存储区类型无关可以采用线性存储）
			- 元素往 pages 方向发展出现类别元素之间相互引用并且聚沙成树（图）化零为整。这里有唯一相对重要的点，单个元素很强组合很好。
			- Tree（索引笔记列表树）+block（分区，中图法分类减少心智压力）
			- 索引树的存储区：顺序存储（首字母和时间线）和中图存储。比如文件系统目录中是按时间 timestore 的顺序存储
- 最初的问题——提前分类不好用
	- 最初是使用印象笔记和 OneNote，这两种结构化的笔记需要在一开始就订立一个框架并承诺这个框架在未来不会发生大的改变，就像学科知识一样。
	- 但问题是，个人笔记会随着时间的流逝而缓慢发生生长，就如树木一样，不断蜕变。这种先固定框架的笔记前期定好结构后后期就没有很大的修改空间了。要修改就要重构
	- 那么我的问题是除了这种层级组织的结构，还有什么其它的结构可以用来组织笔记？
- 追问，什么才是一个好的笔记系统？
	- 第一个问题，怎么组织？除了一开始的固定框架，金字塔式的中图法还有啥？
		- 当数量超过管理能力精力范围时？
		- 当结构成为限制笔记生长时？
	- 第二个问题，怎么检索？有哪些检索方式，如果有关联如何检索关联？？
		- 这依赖于第一个问题，什么样子的组织形式导致了笔记该如何检索
		- 通用的检索方式，ai、笔记软件的搜索、对于有组织系统的检索
	- 第三个问题，如何关联？除了双链，还有关联方式？
		- 由第一个第二个问题引出，笔记之间是需要关联的，正如论文或者软件包一样，是需要相互引用相互依赖的。
	- 第四个问题，笔记是啥？笔记的本质是啥，构成笔记的文字语言的逻辑结构有啥？
		- 笔记的作用、层次、功能？
		- 语言在笔记中的作用是啥？涉及到知识必然设计到语言以及逻辑等方面浅显的知识
		- AI 在笔记中是出于什么角色？
		- 是否触及到人脑的学习和记忆功能？
	- 所以一个好的笔记系统是要有合理的组织方式、可以高效的检索信息、并且可以快速准确的关联笔记、让人们了解笔记的本质更方便记笔记
